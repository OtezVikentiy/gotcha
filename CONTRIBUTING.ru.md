# Участие в разработке Gotcha

Спасибо за интерес к улучшению Gotcha. Этот документ описывает настройку
окружения для разработки, тестирование и принятые в проекте соглашения по коду.

## Требования

- Go 1.26+
- Docker + Docker Compose (для PostgreSQL, ClickHouse и интеграционных тестов)

## Настройка окружения для разработки

```bash
# gitflic (основной, гарантированный анонимный HTTPS)
git clone https://gitflic.ru/project/otezvikentiy/gotcha.git
# GitHub (зеркало, если опубликовано)
git clone https://github.com/OtezVikentiy/gotcha.git
# Контрибьюторы с SSH-доступом могут использовать:
# git clone git@gitflic.ru:otezvikentiy/gotcha.git
cd gotcha
docker compose up -d
```

Это собирает и запускает приложение плюс PostgreSQL и ClickHouse. Шаги первого
запуска (создание bootstrap-администратора, подключение SDK) см. в
[разделе «Быстрый старт» README](README.ru.md#быстрый-старт).

Полезные `make`-цели (полный актуальный список — `make help`):

| Цель | Что делает |
|---|---|
| `make up` / `make down` | Запуск/остановка compose-стека. |
| `make up-rebuild` | Пересобрать образ и принудительно пересоздать контейнеры. |
| `make logs` / `make logs-all` | Следить за логами приложения / логами всех контейнеров. |
| `make health` | Curl `/healthz` на запущенном приложении. |
| `make app-connect` | Зайти в shell контейнера приложения. |
| `make db-connect` | `psql` в PostgreSQL. |
| `make ch-connect` | `clickhouse-client` в ClickHouse. |
| `make run` | `go run ./cmd/gotcha` (нужен запущенный `make up` для PG/CH — см. раздел README «Сборка из исходников», так как compose-файл по умолчанию не публикует порты PG/CH на хост). |
| `make go-build` | Собрать бинарник `gotcha`. |
| `make templ` | Перегенерировать `*_templ.go` из `.templ`-исходников. |
| `make fmt` | `gofmt` всех Go-исходников. |
| `make vet` | `go vet ./...`. |
| `make tidy` | `go mod tidy`. |
| `make check` | `fmt` + `vet` + `test-short` — запускать перед коммитом. |

## Запуск тестов

**Гоняйте тесты щадяще**: набор тестов этого репозитория использует
`testcontainers-go` для поднятия реальных контейнеров PostgreSQL/ClickHouse под
интеграционные тесты. Запуск всего набора с неограниченным параллелизмом может
поднять множество контейнеров сразу и исчерпать память/CPU машины (или Docker
daemon). Всегда запускайте:

```bash
nice -n 19 go test -p 2 ./...
```

— `nice -n 19` понижает приоритет запуска относительно другой работы, а `-p 2`
ограничивает, сколько тестовых бинарников (то есть сколько пакетов, каждый из
которых потенциально поднимает свои контейнеры) идут параллельно. Не запускайте
голый `go test ./...` или `go test -p <N>` с большим `N` в этом репозитории.

Для быстрой итерации без контейнеров — только юнит-тесты:

```bash
go test ./... -short -count=1
```

(именно это запускают `make test-short` и `make check`). Простая цель Makefile
`make test` запускает весь набор через `go test ./... -count=1 -timeout 1800s`
без ограничителя `nice -p 2` — предпочитайте приведённую выше форму
`nice -n 19 go test -p 2 ./...`, когда сами запускаете весь набор.

## Работа с шаблонами templ

Веб-интерфейс рендерится на сервере с помощью [templ](https://templ.guide).
`.templ`-файлы компилируются в `*_templ.go`. **Никогда не правьте руками
`*_templ.go`-файл** — он сгенерирован и будет перезаписан. После изменения
`.templ`-файла запустите:

```bash
make templ
```

и закоммитьте и `.templ`-исходник, и перегенерированный `*_templ.go`-вывод.

## Content Security Policy / никакого inline JS или CSS

Gotcha отдаёт строгий CSP (`default-src 'self'; base-uri 'none';
form-action 'self'; frame-ancestors 'none'`) без `unsafe-inline` и без inline
`<script>`/`<style>` или атрибутов `style="..."`. Вся интерактивность должна
достигаться нативными механизмами HTML/CSS — прежде всего `<details>`/
`<summary>` для раскрывающихся виджетов и CSS `:target` для состояния, которое
должно сохраняться без JavaScript, — плюс htmx для всего, что действительно
требует обращения к серверу. Не вводите inline-обработчики событий, inline-стили
или новые источники скриптов; если изменение кажется требующим этого, сначала
поищите способ сделать то же самое только через CSS или htmx.

## i18n (паритет каталогов RU/EN)

Строки интерфейса берутся из JSON-каталогов под `internal/i18n/locales/` для
`ru` и `en`. Каждый ключ, добавленный в один каталог, должен быть добавлен и в
другой — каталоги должны оставаться в паритете (сообщение, присутствующее только
в одной локали, для другой молча откатится к сырому ключу). Добавляя или изменяя
пользовательские строки, обновляйте `ru.json` и `en.json` в рамках одного
изменения.

## Соглашения по коммитам / PR

- Держите коммиты сфокусированными; пишите сообщения коммитов, объясняющие
  *почему*, а не только *что*.
- По возможности используйте conventional-префиксы (`feat:`, `fix:`, `docs:`,
  `refactor:`, `test:`) — см. существующий стиль в `git log`.
- Запускайте `make check` (или как минимум `gofmt`, `go vet ./...` и тесты,
  относящиеся к вашему изменению) перед открытием PR.
- Если ваше изменение затрагивает `.templ`-файлы, убедитесь, что
  перегенерированные `*_templ.go`-файлы включены в тот же коммит.
- Если ваше изменение добавляет или меняет пользовательский текст, обновите оба
  i18n-каталога (см. выше).
- Опишите «почему» в описании PR; свяжите его с относящейся к нему задачей.
